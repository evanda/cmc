# Going real: Supabase setup runbook

Stand up a live backend for one church instance (single-tenant per deployment,
plan §7.6) and validate the parts that demo mode can't: **RLS, auth, and
Storage**. ~15–20 min. Everything in the app has so far been verified only
against the demo store + a plain-Postgres migration check — this is the step
that makes it real.

## 0. Prereqgs
- A Supabase account. Either the **CLI + Docker** (local stack) or a **hosted
  project** works; hosted is simplest for a first real instance.
- This repo checked out, `pnpm install` done.

---

## 1. Create the project & get keys
1. Create a new project at app.supabase.com (pick a region, set a DB password).
2. Project → **Settings → API**: copy
   - **Project URL** → `VITE_SUPABASE_URL` / `SUPABASE_URL`
   - **anon public** key → `VITE_SUPABASE_ANON_KEY`
   - **service_role** key → `SUPABASE_SERVICE_ROLE_KEY` (server/CLI only — never ship to the client)

```bash
cp .env.example .env      # then fill the four values above. Never commit .env.
```

## 2. Apply the schema (12 migrations)
**Option A — CLI (recommended):**
```bash
pnpm dlx supabase login
pnpm dlx supabase link --project-ref <your-ref>
pnpm dlx supabase db push          # applies supabase/migrations/* in order
```
**Option B — Dashboard:** SQL Editor → paste each `supabase/migrations/00*.sql`
in numeric order and run. (Migrations `0005`/`0007` create the Storage buckets;
on hosted Supabase the `storage` schema exists so they run for real, unlike the
local plain-PG check where they no-op.)

Confirm: Table editor shows `org_settings`, `users`, `buildings`, `floors`,
`locations`, `assets`, `work_orders`, `work_requests`, `vendors`, `pois`,
`pm_schedules`, … and **RLS is enabled** on each (lock icon).

## 3. Create the first admin
New signups land as `requester` (the `handle_new_auth_user` trigger). Promote
yourself once:
1. Run the app against Supabase (step 5) and **sign up** with your email, OR
   create a user in Dashboard → Authentication → Users.
2. SQL Editor:
   ```sql
   update public.users set role = 'admin' where email = 'you@example.org';
   ```

## 4. Seed content
- Org identity + a sample campus:
  ```bash
  pnpm db:seed bigcampus        # uses SUPABASE_URL + SUPABASE_SERVICE_ROLE_KEY
  ```
- Or load the real **Midway PCA** facility: the data lives in
  `map-data/facilities/midwaypca/` and `apps/web/src/data/midwaypca.json`
  (generated by `scripts/gen-midway-demo.mjs`). A DB importer for it is tracked
  in **#14** — until then `db:seed` covers buildings/floors/assets.

## 5. Run the app against the backend
The web app auto-loads the root `.env` (Vite `envDir`); just make sure
`VITE_DEMO` is UNSET so it uses the real client:
```bash
pnpm --filter @cmc/web dev          # http://localhost:5173
```
Sign in with your admin user. The header should show your org name from
`org_settings`. (More dev-run detail — demo mode, restart, ports — in
[`CONTRIBUTING.md`](../CONTRIBUTING.md).)

## 6. Validate RLS & Storage (the point of all this — issue #17)
Create a couple of test users (Dashboard → Auth) and set their roles via SQL
(`technician`, `requester`, `trustee`). Then verify:

| Role | Should be able to | Should be blocked from |
|------|-------------------|------------------------|
| admin | everything incl. buildings/users/PM config | — |
| technician | create/edit assets, work orders, log work, PM | editing users, org settings |
| requester | submit a request; see **only their own** requests | assets/WO/config writes |
| trustee | read dashboards/reports | all writes |

- **Storage**: open an asset → add a photo. Confirm it uploads to the
  `asset-photos` bucket (Dashboard → Storage) and renders. Repeat a before/after
  photo on a work order (`work-order-photos`). Public read, staff-only write.
- **Signup→profile**: a brand-new signup appears in `public.users` as
  `requester`.

If any boundary is wrong, the policy lives in the matching migration
(`0002_rls.sql` for core; `0004`/`0006`/`0008`/`0009`/`0010`/`0011` per table).

## 7. Deploy (later)
- Web → Vercel project with the two `VITE_SUPABASE_*` env vars.
- The daily PM job + expiry sweep (issue #18) and user email invites (#13) are
  Edge Functions / `pg_cron` — set up once the basics above are confirmed.

---

### Troubleshooting
- **Login works but no data / RLS denies everything** → your user has no
  `public.users` row or wrong role; check step 3.
- **Photo upload 403** → bucket missing or you're not staff; confirm `0005`/`0007`
  ran on hosted (not the local no-op) and your role is admin/technician.
- **`db:seed` fails** → `SUPABASE_URL` / `SUPABASE_SERVICE_ROLE_KEY` not set in
  the shell env.
