## The data-access boundary (`apps/web/src/lib/`)

All reads/writes go through a single **`DataSource`** interface
(`datasource.ts`) with two implementations:

- **`supabaseDataSource`** â€” the real backend, RLS-enforced, typed via
  `packages/shared`'s hand-authored `Database` type.
- **`demoDataSource`** (`demo.ts`) â€” an in-memory store seeded with sample data.

`ds` is chosen at module load from `isDemo` (`supabase.ts`). React Query hooks
in `queries.ts` call `ds.*` and never touch Supabase directly, so the UI is
agnostic to which backend is live.

### Demo mode (offline, no backend)

`VITE_DEMO` selects an offline dataset and a fake signed-in admin
(`AuthProvider`), so the whole app renders with **no Supabase**:

- `VITE_DEMO=1` / `sample` â€” a fictional multi-building "Sample Campus".
- `VITE_DEMO=midway` â€” the real **Midway PCA** campus, reseeded from
  `apps/web/src/data/midwaypca.json` (generated by
  `scripts/gen-midway-demo.mjs` from `map-data/facilities/midwaypca/`).

This is what powers the screenshot harness and lets the UI be reviewed without
provisioning a database. It's gated by env, so production builds tree-shake it
out.

## Database & migrations (`supabase/`)

The relational schema (plan Â§6) is built up by numbered migrations under
`supabase/migrations/` â€” **never edit an applied migration in place**; add a new
one. Conventions:

- Every table carries `id uuid`, `created_at`, `updated_at` (touch trigger),
  `created_by default auth.uid()`, and a soft-delete `deleted_at`.
- Enums mirror `packages/shared/src/types/enums.ts`.
- **Single-tenant**: there is no `facility_id` threaded through tables; church
  identity is the single `org_settings` row.

### RLS model (plan Â§7.5)

RLS is on for every table. `SECURITY DEFINER` helpers `is_admin()` / `is_staff()`
read the caller's role from `public.users`. Pattern: **read for `authenticated`,
write for staff** (admin/technician); config tables (buildings/floors/locations/
categories/org_settings) are admin-write; requesters see only their own work
requests.

### Verifying migrations without Docker

Supabase's local stack needs Docker. When it isn't available, migrations are
verified against a plain Postgres 16 using `scripts/_local-pg-auth-shim.sql`
(creates the `auth` schema + `anon`/`authenticated`/`service_role` roles +
`auth.uid()`), then applying `supabase/migrations/00*.sql` in order. Storage
bucket migrations are guarded to no-op where the `storage` schema is absent.

## Spatial / map (plan Â§5)

One MapLibre GL JS map, one geographic coordinate system. `MapView.tsx` renders
building footprints + areas + equipment POIs from GeoJSON and a **level
switcher** (`Site / B1 / 1 / 2`) that filters POIs by floor. No external
tiles â€” a neutral basemap keeps it fully offline.

Data flow: `map-data/facilities/<id>/*.geojson` â†’ copied to
`apps/web/public/facilities/<id>/` â†’ fetched at runtime by `MapView`. (Backing
the map with the DB `pois`/`floors` tables and linking POIs to assets is
tracked in issue #5.)
